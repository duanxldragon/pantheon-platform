# Frontend Docs AGENTS

> 本文件补充 `../AGENTS.md` 与 `../../AGENTS.md`，只约束 `frontend/docs/` 下的默认行为。

## 1. 文档边界

- 这里只写前端实现视角，不重复平台级设计。
- 这里回答“前端状态怎么流转、界面怎么挂载、交互怎么承接”。
- 平台级业务规则继续放在 `../../docs/`。
- 页面源码与组件细节仍以 `src/` 为准，这里只保留结构化说明。

## 2. 写文档前先判断

- 这是前端实现问题，还是平台设计问题？
- 这是总览问题，还是认证 / 系统 / 租户专题问题？
- 现有文档是否已经能承载这次更新？

## 3. 默认写法

- 先写应用初始化，再写状态协作，再写挂载方式。
- 先写 Store、菜单、权限、视图之间的关系，再写页面层表现。
- 避免把组件源码逐段翻译成文档。
- 对变化频繁的 UI 细节，只给入口和约束，不在文档里堆视觉描述。

## 4. 明确禁止

- 不要把 `frontend/docs/` 写成页面操作手册。
- 不要把平台级规则和前端实现规则混写。
- 不要在多个文档重复维护同一份状态流转说明。
- 不要为了“完整”而复制大段 TSX 代码。

## 5. 推荐分工

- `FRONTEND_GUIDE.md`：前端工程总览、初始化链路、状态体系、接入方式
- `docs/FRONTEND_DOCS_INDEX.md`：前端文档入口
- `docs/auth/`：认证与会话相关的前端实现
- `docs/system/`：系统管理模块的前端实现
- `docs/tenant/`：租户初始化与租户管理的前端实现

## 6. 更新原则

- 改登录态、刷新链路、菜单挂载、动态视图、Store 分工时，要同步更新文档。
- 改通用 API、拦截器、全局错误处理时，也要同步更新相关文档。
- 入口文档保持短，细节拆到专题文档。

## 7. 提交前

- 说明改的是总览还是专题实现文档
- 说明是否影响初始化链路、状态协作、菜单挂载或权限判断
- 说明是否还有其他前端文档需要后续同步
