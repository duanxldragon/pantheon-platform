# 系统初始化分层说明

> 本文只说明前端登录后初始化、refresh 重建和系统快照之间的分层关系。  
> 它不重复认证规则，也不重复系统管理业务模型。

## 1. 为什么需要分层说明

当前前端已经具备统一的登录后恢复链路，但文档中容易把下面几件事混在一起：

- 登录成功后的认证恢复；
- 租户上下文检查；
- 系统菜单与管理数据初始化；
- 页面进入后的按需加载。

把这些层次拆开后，才能更准确地讨论性能、职责边界和后续优化。

---

## 2. 当前实际初始化入口

当前最关键的入口不是页面各自发请求，而是认证链路中的统一恢复入口：

- `authStore.reloadAuthorization()`
- `authStore.checkTenantStatus()`
- `authStore.refreshTenantContext()`
- `systemStore.initialize()`

其中：

- `refreshTenantContext()` 负责把认证、租户、系统快照串起来；
- `systemStore.initialize()` 负责系统管理相关的大快照重建。

---

## 3. 当前三层初始化模型

可以把当前前端运行时理解为三层：

### 3.1 第一层：会话恢复层

目标是回答：

- 当前用户是谁；
- 当前 access token 是否已续签；
- 当前权限集合是否需要重拉。

这层主要由 `authStore` 承担。

### 3.2 第二层：租户上下文层

目标是回答：

- 当前是否启用多租户；
- 当前租户是否已初始化；
- 当前租户是否允许进入正式业务界面。

这层也是由 `authStore` 协调，但它更关注租户状态，而不是页面数据本身。

### 3.3 第三层：系统快照层

目标是回答：

- 当前菜单树是什么；
- 系统管理页所需的基础列表是否已具备；
- 系统设置、日志、角色等缓存是否需要重建。

这层主要由 `systemStore.initialize()` 承担。

---

## 4. 当前 `systemStore.initialize()` 的范围

当前 `systemStore.initialize()` 会并发拉取：

- 用户列表；
- 部门列表；
- 角色列表；
- 当前用户菜单；
- 岗位列表；
- 操作日志；
- 系统设置。

这意味着它更接近“系统管理快照加载器”，而不是“仅为壳层首屏服务的轻量初始化器”。

这样设计的好处是：

- 登录后系统管理模块可以较快进入可操作状态；
- refresh 后可以一次性重建大部分系统级快照；
- 菜单、角色、设置等依赖能统一恢复。

当前代价也很明确：

- 壳层初始化和系统管理数据加载耦合较紧；
- 非系统管理页面也会承担一部分初始化成本；
- 后续如果模块继续增多，初始化负载会继续上涨。

---

## 5. 当前推荐理解方式

为了避免误解，建议这样理解现状：

- `authStore` 不是只管 token，而是“会话恢复协调者”；
- `systemStore` 不是纯菜单仓库，而是“系统管理快照仓库”；
- `uiStore` 不参与数据恢复，只负责工作台标签与激活视图。

这样划分后，职责边界会更清楚：

- 会话问题优先看 `authStore`
- 菜单与系统快照问题优先看 `systemStore`
- 标签页和动态视图切换问题优先看 `uiStore` 与动态视图运行时

---

## 6. 后续可演进方向

以下是建议，不代表当前已经实现：

### 6.1 拆成“最小壳层初始化”

优先只加载：

- 当前用户；
- 当前权限；
- 当前菜单；
- 租户就绪状态。

### 6.2 把系统管理数据改为按需进入

例如把以下数据延后到具体页面再拉取：

- 用户列表；
- 部门列表；
- 岗位列表；
- 操作日志；
- 系统设置。

### 6.3 保持统一恢复入口不变

即使后续继续拆分，也建议保留：

- refresh 统一入口；
- 认证恢复统一入口；
- 租户状态检查统一入口。

不要回到“每个页面各自补救登录态”的模式。
