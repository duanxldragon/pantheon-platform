# Docs AGENTS

> 本文件补充 `../AGENTS.md`，只约束 `docs/` 目录下的默认行为。

## 1. 文档目标

- `docs/` 只写平台级设计，不写前后端细节实现。
- 这里回答“平台怎么设计、模块怎么协作、边界怎么划分”。
- 代码级细节应分别落到 `backend/docs/` 或 `frontend/docs/`。

## 2. 写文档前先判断

- 这是平台级设计问题，还是实现细节问题？
- 这是在补入口文档，还是在补专题文档？
- 现有文档是否已经覆盖，只是需要更新，而不是新增一份重复文档？

## 3. 明确禁止

- 不要把 `docs/` 写成接口注释汇总。
- 不要把后端目录、函数、实现细节大段搬进平台文档。
- 不要在多个文档重复维护同一套规则。
- 不要为了“写全”而堆砌大段代码片段。
- 不要保留明显过时的流程说明。

## 4. 默认写法

- 先写边界，再写链路，再写入口。
- 先回答“是什么、为什么、怎么协作”，再回答“去哪里看实现”。
- 优先使用短标题、短列表、短段落。
- 避免和 `ESLint`、`Prettier`、测试规则重复。
- 文档文件默认保持 `UTF-8`；出现中文乱码时，先检查编码，不要直接重写内容。

## 5. 文档分工

- `README.md`：项目入口与总体定位
- `docs/`：平台级设计
- `backend/BACKEND_GUIDE.md`：后端工程总览
- `frontend/FRONTEND_GUIDE.md`：前端工程总览
- `backend/docs/`：后端专题实现
- `frontend/docs/`：前端专题实现

## 6. 更新原则

- 改架构、主链路、模块边界时，要同步更新文档。
- 新增文档前，优先判断能不能并入现有入口或专题文档。
- 文档应保持简洁；超长时拆专题，而不是把入口文档写成百科。

## 7. 提交前

- 说明更新的是哪类文档
- 说明文档边界是否发生变化
- 说明是否还有相关文档需要后续同步
