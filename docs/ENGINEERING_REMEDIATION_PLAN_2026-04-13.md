## 2. 执行目标

本轮整改聚焦 4 个核心目标：

1. 建立前端核心运行时白盒测试基线；
2. 把正式 E2E 回归纳入标准 CI；
3. 补齐 API Key 高安全能力；
4. 把规范逐步转化为 Review / CI / Release 的默认执行动作。

## 4. 分阶段执行计划

## 阶段 A：修复工程红线

### 目标

先解决“规范已存在，但最关键执行缺口仍未补上”的问题。

### A-1 技术基线统一

- [ ] 对齐 `backend/go.mod` 与后端 CI 的 Go 版本
- [ ] 确认本地开发、CI、构建机使用一致的 Go 主版本
- [ ] 在后端文档或 CI 注释中明确版本基线

**完成定义**

- [ ] 本地与 CI Go 版本一致
- [ ] 后端 CI 不再出现因版本偏差导致的不一致问题

### A-2 前端白盒测试最小集

#### 范围

- [ ] `authStore`
- [ ] `systemStore`
- [ ] `uiStore`
- [ ] `useViewManager`
- [ ] `RouteGuard`
- [ ] `QueryAccessBoundary`

#### 最低必测点

- [ ] 登录态与租户未初始化分流
- [ ] 权限判断与角色判断
- [ ] 菜单变化后标签页回收
- [ ] 不可访问页面的 fallback 逻辑
- [ ] 查询权限丢失后的自动收起与回跳

**完成定义**

- [ ] `frontend/src/` 下形成正式测试文件
- [ ] 测试不依赖临时页面或人工环境
- [ ] 覆盖工作台主运行时核心规则

### A-3 E2E 资产治理

- [ ] 盘点 `frontend/tests/` 中正式用例与临时用例
- [ ] 归档或删除 `tmp-*` 规格
- [ ] 明确“长期保留的正式主链路用例”
- [ ] 明确“临时排障脚本”的存放约定

**完成定义**

- [ ] E2E 目录中不再混入临时用例
- [ ] 主链路测试集可以被明确点名执行

### A-4 正式 E2E 纳入 CI

- [ ] 选定主链路 Playwright 套件
- [ ] 在前端 CI 或独立 workflow 中加入 E2E job
- [ ] 明确 E2E 的环境变量与测试数据要求
- [ ] 明确失败截图、trace、report 的归档策略

**完成定义**

- [ ] CI 可自动执行正式 E2E
- [ ] 失败后能定位到具体链路
- [ ] 有可下载的报告产物

## 9. 最终建议

本轮整改不建议继续扩写大量新规范。

更高价值的做法是：

- 先把现有规范执行起来；
- 再把执行动作自动化；
- 最后再继续细化规范。

对当前仓库而言，最重要的不是“再补一份文档”，而是让以下 3 件事成为默认事实：

1. 核心运行时有白盒测试；
2. 主用户旅程有正式 E2E 门禁；
3. 高风险能力有更强安全约束与更明确 Review 机制。

做到这一步，后续无论是团队协作还是 AI 辅助开发，都会顺很多。
