# Pantheon Platform 功能缺口系统性评估（2026-04-10）

## 1. 评估目标与方法

本评估聚焦“已声明能力”和“代码可用能力”之间的缺口，目标是识别当前代码库中的**缺失功能**，并给出分级建议。

评估方法：

1. 以 `README.md` 与平台级文档中的能力清单作为预期基线。
2. 对照后端 `router/handler/service` 与前端 `views/api/stores` 的落地实现。
3. 重点标记三类问题：
   - 明确 TODO / not implemented；
   - 有底层能力但无 API / UI 入口；
   - 前后端能力不对齐。

---

## 2. 结论总览

### 2.1 总体结论

当前平台主链路（`auth / tenant / system`）可运行，但仍存在 6 个高价值功能缺口：

- **P0（建议尽快排期）**：2 项
- **P1（建议近期补齐）**：3 项
- **P2（建议中期治理）**：1 项

### 2.2 缺口清单（按优先级）

| 优先级 | 功能缺口 | 当前状态 | 影响 |
| :-- | :-- | :-- | :-- |
| P0 | 对象存储（S3/OSS）上传/删除未实现 | Provider 仅占位，调用返回 not implemented | 生产环境无法使用对象存储上传头像/附件等 |
| P0 | 租户备份能力缺少 API/页面入口 | 有 `backup_service` 但未对外暴露路由/前端流程 | 运维无法在平台内执行租户备份/恢复 |
| P1 | 通知模板与发送链路缺少完整管理 UI | 前端已有 API 封装，但缺少模板 CRUD 与发送编排界面 | 通知模块可维护性与运营效率不足 |
| P1 | 通知异步任务处理缺少调度入口 | 后端仅提供手动 `jobs/process` 接口 | 任务积压与可靠性风险，依赖人工触发 |
| P1 | 动态模块加载能力处于占位状态 | Go/JS Loader 与反射创建均未实现 | 插件化扩展不可用，模块生态受限 |
| P2 | 自动化质量门禁仍偏弱（前端测试规模不足） | 已有测试脚手架，但覆盖关键链路不足 | 重构和发版回归风险偏高 |

---

## 3. 关键缺口详细说明

### 3.1 P0-1：对象存储 S3/OSS 未落地

**发现**

- `S3StorageProvider.Upload` / `Delete` 仍是占位实现，直接返回未实现错误。

**风险**

- 一旦部署配置切到 `storage.provider=s3`，上传能力不可用，影响头像、文件、通知附件等依赖存储的场景。

**建议**

- 第一阶段：先补全最小可用上传/删除 + 错误分类（鉴权失败、桶不存在、网络错误）。
- 第二阶段：补回调重试、预签名 URL、分片上传（按文件大小阈值）。

### 3.2 P0-2：租户备份服务“有内核，无入口”

**发现**

- `tenant` 模块存在 `backup_service.go`（含创建、列举、删除、恢复等能力）。
- 但当前租户路由未暴露 `/backups` 相关 API，前端也无对应页面/流程。

**风险**

- 关键灾备能力只能靠外部脚本或直接操作，不利于平台化交付和标准运维。

**建议**

- 先补最小闭环：`create/list/delete/restore` 四个租户备份 API。
- 再补前端运维页（含权限码、操作审计、二次确认）。

### 3.3 P1-1：通知模板/发送编排 UI 不完整

**发现**

- 前端 `notification_api` 已有模板 CRUD 与 `sendFromTemplate` 接口封装。
- 当前 `NotificationCenter` 主要承载收件箱和模板列表展示，缺少模板新增/编辑/删除与发送编排的完整交互界面。

**风险**

- 运营侧无法在平台内闭环管理模板版本与发送任务，需依赖脚本或临时接口调试。

**建议**

- 增加模板编辑抽屉（变量校验、预览、版本备注）。
- 增加发送向导（选择模板 -> 变量渲染预览 -> 接收人范围 -> 发送策略）。

### 3.4 P1-2：通知异步任务缺少系统调度

**发现**

- 后端存在 `POST /api/v1/notifications/jobs/process` 手动处理接口。
- 代码中未看到常驻调度器/定时任务注册入口。

**风险**

- 高峰期或人工遗漏时，消息发送可能延迟；任务重试策略依赖人工触发。

**建议**

- 引入后台 worker（ticker 或队列消费模型），把 `process` 从“人工触发”改为“系统自动拉取 + 失败退避重试”。

### 3.5 P1-3：动态模块加载仍是规划态

**发现**

- `module_registry.go` 中 Go 插件加载、JS 模块加载、反射实例化均为 TODO 占位。

**风险**

- “可插拔模块”无法落地，后续生态扩展能力受限。

**建议**

- 若短期不做插件生态，建议在文档里降级为“规划中”，避免与当前能力预期不一致。
- 若要落地，先实现“受控白名单 + 本地插件目录 + 签名校验”最小方案。

### 3.6 P2-1：前端测试覆盖关键链路不足

**发现**

- 前端已接入 `vitest` / `playwright`，但当前测试文件数量和业务覆盖仍有限。

**风险**

- 登录刷新链路、菜单动态挂载、权限变更等核心流程回归风险高。

**建议**

- 先覆盖 3 条红线流程：登录（含 2FA）/ token refresh / 权限变更后页面重建。

---

## 4. 建议落地顺序（两周节奏）

### Sprint A（优先恢复可用性）

1. 完成 S3/OSS 最小可用实现（上传、删除、错误处理）。
2. 开放租户备份 API（含审计与权限码）。

### Sprint B（补运营与可靠性）

1. 增加通知模板管理与发送向导 UI。
2. 引入通知任务后台调度器。
3. 补充关键链路自动化测试（认证刷新 + 动态菜单）。

---

## 5. 风险与后续建议

- 若近期要推进私有化交付，建议把 **S3/备份** 作为交付阻断项。
- 若近期要推进 SaaS 运营，建议把 **通知调度 + 模板管理** 作为运营阻断项。
- 建议在 `docs/` 增加“能力状态矩阵（已实现 / 部分实现 / 规划中）”，降低文档与代码预期偏差。
