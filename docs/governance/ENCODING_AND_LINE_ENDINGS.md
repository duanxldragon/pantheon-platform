# 文档编码与换行规范

> 这份文档用于约束 Pantheon Platform 仓库中的文本编码与换行策略。  
> 目标是避免中文乱码、跨平台换行抖动和无意义的文本差异。

## 1. 默认规则

- 所有文本文件默认使用 `UTF-8`
- 大多数源码、配置、文档文件统一使用 `LF`
- Windows 脚本文件保留 `CRLF`

当前仓库中的长期兜底配置为：

- `.editorconfig`
- `.gitattributes`

## 2. 各类文件约定

统一使用 `LF` 的常见文件包括：

- `*.md`
- `*.go`
- `*.ts`
- `*.tsx`
- `*.js`
- `*.json`
- `*.yml`
- `*.yaml`
- `*.css`
- `*.html`
- `*.sql`

保留 `CRLF` 的 Windows 脚本包括：

- `*.ps1`
- `*.bat`
- `*.cmd`

二进制文件不得按文本处理，例如：

- 图片
- 字体
- 压缩包
- PDF

## 3. 为什么这样约束

这样做主要是为了避免：

- GitHub 或编辑器中出现中文乱码
- Windows 与 Linux/macOS 团队成员之间反复产生换行差异
- 提交中混入大批无意义的全文变更
- 脚本因换行不符合宿主平台习惯而出现执行问题

## 4. 编辑器与 Git 的职责分工

- `.editorconfig` 负责约束编辑器保存行为
- `.gitattributes` 负责约束 Git 的文本归一化与检出行为

两者应同时存在，不建议只依赖其中一个。

## 5. 团队协作要求

- 新建文本文件时，默认按 `UTF-8` 保存
- 修改 Markdown、代码、配置文件时，不要顺手改整文件换行
- 发现中文显示异常时，先检查文件实际编码，而不是直接重写内容
- 不要把临时导出的日志、测试产物、构建产物提交进仓库

## 6. 排查建议

当出现乱码或换行异常时，优先按以下顺序排查：

1. 检查文件是否为 `UTF-8`
2. 检查编辑器当前打开编码
3. 检查 `.editorconfig` 是否生效
4. 检查 `.gitattributes` 是否覆盖对应文件类型
5. 检查本地 Git 配置是否额外改写了换行行为

## 7. 当前仓库基线

当前仓库的受版本控制 Markdown 文档已经统一为 `UTF-8`。

后续如果新增新的文本文件类型，需同步补充：

- `.editorconfig`
- `.gitattributes`
- 本文档（如有必要）
