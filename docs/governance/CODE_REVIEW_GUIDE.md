# Code Review 规范

> 本文定义 Pantheon Platform 的标准 Code Review 口径。  
> 目标不是拉长流程，而是保证架构边界、质量基线和长期可维护性。

## 1. Review 目标

每次 Review 都应回答四个问题：

1. 改动是否落在正确架构边界；
2. 是否破坏认证、租户、权限或运行时链路；
3. 是否有与改动相称的测试与文档；
4. 是否把临时修补包装成长期方案。

## 2. Review 顺序

建议按下面顺序看：

1. 需求与影响范围；
2. 设计边界；
3. 目录落点；
4. 核心链路；
5. 错误与异常路径；
6. 数据与安全；
7. 测试；
8. 文档。

## 3. 平台级必查项

### 3.1 架构

- 是否遵守 `docs/design/SYSTEM_ARCHITECTURE.md`；
- 是否错误地把平台设计、前端实现、后端实现混到一起；
- 是否引入了重复的状态、权限、菜单、主题或请求链路。

### 3.2 多租户

- 是否绕开租户上下文；
- 是否把租户级数据错误落到主库；
- 是否遗漏租户迁移器、连接池或生命周期联动；
- 是否存在跨租户读取/写入风险。

### 3.3 认证与授权

- 是否破坏 `JWT + Refresh Token + Redis + revoked/version`；
- 是否考虑到角色、菜单、权限变更后的会话刷新；
- 是否遗漏安全事件强制失效；
- 是否只在前端做权限控制而未在后端兜底。

### 3.4 审计与高风险动作

- 删除、停用、重置密码、踢会话等动作是否可追踪；
- 批量操作是否具备原子性或明确失败语义；
- 高风险动作是否存在明确确认与反馈。

## 4. 前端 Review 清单

- 是否遵守 `frontend/docs/FRONTEND_CODE_STANDARDS.md`；
- 页面是否绕过 API 层直接散落写请求；
- 是否误把模块私有逻辑塞进 `shared/`；
- 是否破坏 `authStore / systemStore / uiStore / React Router` 分工；
- 涉及菜单、权限、标签页、动态视图时，是否同时检查运行时链路；
- 是否硬编码文案、权限码、路由或主题色值；
- 是否遵守 `i18n`、编码、UI 规范。

## 5. 后端 Review 清单

- 是否遵守 `backend/docs/BACKEND_CODE_STANDARDS.md`；
- `handler / service / dao / router` 职责是否混乱；
- 是否把编排逻辑塞进 `handler` 或 `main.go`；
- 是否遗漏事务边界、错误包装、统一响应；
- 是否直接编辑生成产物；
- 是否遗漏迁移、初始化、默认数据或工具链同步。

## 6. 测试 Review 清单

- 是否补了与改动相称的验证；
- 白盒测试是否覆盖关键分支与失败路径；
- E2E 是否覆盖关键用户链路；
- 仅有 happy path 时，是否遗漏边界/异常态；
- 是否记录未执行项和原因。

## 7. 文档 Review 清单

- 是否同步更新入口文档或专题文档；
- 文档中的术语、路径、链路是否与代码一致；
- 是否把建议写成现状；
- 是否引入重复维护的规则。

## 8. 结论分级

### 必须修改

- 架构边界错误；
- 租户隔离风险；
- 权限/认证链路缺口；
- 高风险写操作缺少保护；
- 没有最小必要测试。

### 建议修改

- 命名、结构、复用层级不理想；
- 文档缺入口或链接；
- UI / 文案 / i18n 细节不统一；
- 测试粒度偏粗但还能接受。

### 可接受后续处理

- 不影响主链路的增强项；
- 辅助测试补充；
- 非阻塞的样式与交互优化。
