# 工程开发流程

> 本文定义 Pantheon Platform 的标准开发流程。  
> 目标是把“需求 → 设计 → 编码 → Review → 测试 → 交付”串成一条统一主线，减少后续开发中的口径漂移。

## 1. 适用范围

适用于：

- 新功能开发；
- 现有功能增强；
- 缺陷修复；
- 架构调整；
- 文档与测试体系补齐。

## 2. 流程总览

标准流程建议按下面顺序推进：

1. 需求进入；
2. 影响分析；
3. 设计确认；
4. 任务拆解；
5. 编码实现；
6. 自查与 Review；
7. 分层测试；
8. 文档同步；
9. 验收与交付。

## 3. 阶段说明

### 3.1 需求进入

先明确四件事：

- 需求目标是什么；
- 属于哪条主链路：`auth / tenant / system / notification / 壳层`；
- 是新增能力、增强还是修复；
- 是否涉及安全、租户、权限、初始化等高风险边界。

产出建议：

- 一页需求摘要；
- 影响模块清单；
- 初步风险判断。

### 3.2 影响分析

至少判断：

- 数据落点：主库还是租户库；
- 接口边界：是否新增或变更 API；
- 前端边界：是否影响菜单、标签页、动态视图、页面挂载；
- 后端边界：是否影响 handler / service / dao / 中间件；
- 运行时边界：是否影响登录、刷新、租户初始化、会话失效；
- 测试边界：需要补哪些白盒、集成、E2E。

### 3.3 设计确认

设计阶段至少统一：

- 功能边界与非目标；
- 模块协作关系；
- 权限与租户隔离策略；
- UI 承接方式；
- 错误与回滚语义；
- 交付和验收口径。

参考文档：

- `docs/design/SYSTEM_ARCHITECTURE.md`
- `docs/system/UI_DESIGN.md`
- `docs/api/API_DESIGN_STANDARDS.md`
- `docs/testing/TEST_STRATEGY.md`

### 3.4 任务拆解

拆解时优先按“主链路 + 独立写入边界”分组，而不是只按前后端二分。

推荐拆法：

- 平台设计 / 规则；
- 后端实现；
- 前端实现；
- 数据迁移 / 初始化；
- 测试；
- 文档与验收。

建议使用：

- `docs/development/FEATURE_DELIVERY_TEMPLATE.md`

### 3.5 编码实现

实施时遵守：

- 前端：`frontend/docs/FRONTEND_CODE_STANDARDS.md`
- 后端：`backend/docs/BACKEND_CODE_STANDARDS.md`
- Review：`docs/governance/CODE_REVIEW_GUIDE.md`
- 编码：`docs/governance/ENCODING_AND_LINE_ENDINGS.md`

基本要求：

- 不引入第二套机制；
- 不绕过租户上下文或统一鉴权；
- 不把临时方案包装成长期方案；
- 不遗漏高风险动作的确认、审计和失败反馈。

### 3.6 自查与 Review

提交前至少完成：

- 开发者自查；
- 关键链路走查；
- Code Review；
- 风险和未覆盖项记录。

统一口径：

- `docs/governance/CODE_REVIEW_GUIDE.md`
- `docs/governance/SYSTEM_CHECKLIST.md`

进入 GitHub PR 阶段时建议继续细化为：

1. 先开 Draft PR，先对齐范围、风险、主链路；
2. 自查通过后再请求正式 review；
3. review comment 必须区分“必须修改 / 建议修改 / 可后续处理”；
4. 需要补测试、补文档或补截图时，直接在同一 PR 收口；
5. 合并前再次确认 required checks、回滚方案和发布影响。

### 3.7 分层测试

测试执行顺序建议：

1. L0 静态 / 构建检查；
2. L1 白盒测试；
3. L2 集成测试；
4. L3 E2E / 冒烟；
5. L4 安全 / 性能 / 发布前专项验证。

统一口径：

- `docs/testing/TEST_STRATEGY.md`
- `docs/testing/WHITE_BOX_TEST_GUIDE.md`
- `docs/testing/E2E_TEST_PLAN.md`

### 3.8 文档同步

只更新真正被影响的文档：

- 平台级规则进 `docs/`；
- 后端实现进 `backend/docs/`；
- 前端实现进 `frontend/docs/`；
- 阶段性报告或一次性计划不要进入现行规范目录，必要时保留在任务系统、PR 说明或仓库外部记录中。

### 3.9 验收与交付

交付时至少说明：

- 改了什么；
- 为什么这样改；
- 做了哪些验证；
- 还有哪些风险、限制或后续建议。

## 4. 触发矩阵

### 4.1 认证链路改动

必须同时检查：

- 登录 / 刷新 / 退出；
- 会话撤销与版本失效；
- 菜单和权限刷新；
- 前后端错误反馈一致性。

### 4.2 租户链路改动

必须同时检查：

- 初始化状态；
- 数据库连接与迁移；
- 租户停用 / 删除 / 切换；
- 前端引导页与后端租户上下文。

### 4.3 系统管理改动

必须同时检查：

- 角色、权限、菜单联动；
- 数据范围与越权拒绝；
- 工作台挂载与页面承接；
- 主流程回归。

### 4.4 壳层与运行时改动

必须同时检查：

- `authStore`
- `systemStore`
- `uiStore`
- 路由守卫
- 菜单、标签页和动态视图链路

## 5. 完成定义

一个功能可视为“达到交付基线”，至少满足：

- 需求边界清楚；
- 设计路径清楚；
- 改动落在正确目录；
- 已通过自查和 Review；
- 已完成与风险相称的测试；
- 相关文档已同步；
- 风险项和未覆盖项已说明。
