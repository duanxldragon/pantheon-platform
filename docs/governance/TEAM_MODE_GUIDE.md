# Team 模式协作指南

> 本文定义 Pantheon Platform 在开发、排障、重构、联调时的默认协作方式。  
> 目标不是“多开几个代理”，而是把任务拆分、边界判断、并行推进、集成验收固定成一套可复用的方法。

## 1. 适用范围

适用于以下类型工作：

- 缺陷排查与修复；
- 新功能接入；
- 跨模块联调；
- 权限、菜单、初始化、租户链路变更；
- 需要同时改文档、代码、验证的任务。

不适用于以下场景：

- 单文件微调且边界极清晰的改动；
- 纯文案修改；
- 不涉及代码库上下文的通用咨询。

---

## 2. 为什么本仓库需要 Team 模式

Pantheon Platform 不是普通单体后台页面工程，至少有三层复杂性会同时出现：

- 业务主线不是一个，而是 `auth`、`tenant`、`system` 三条主线协作；
- 前端不是传统静态路由后台，而是“菜单 + 标签页 + 动态视图”模型；
- 后端存在 `Master DB`、`Tenant DB`、`Redis`、`Casbin`、租户迁移器等协作链路。

这意味着很多问题不能按“前端一个人修，后端一个人修”粗暴切开。

例如：

- 菜单问题，往往同时涉及前端菜单树、`ViewManager`、角色菜单关系、后端菜单接口；
- 权限问题，往往同时涉及 `authStore`、`systemStore`、刷新链路、Casbin 与中间件；
- 租户初始化问题，往往同时涉及前端引导、后端租户服务、数据库连接管理与默认数据。

因此本仓库更适合按“边界职责 + 主链路”组织协作，而不是只按技术层分工。

---

## 3. 默认团队设计

不是每个任务都需要全部角色，但默认角色池固定如下。

### 3.1 主协调

负责：

- 读需求、定边界、定验收；
- 识别是否涉及 `auth`、`tenant`、`system` 主线；
- 拆解任务，决定哪些可以并行；
- 集成最终改动并统一汇报。

要求：

- 不把关键阻塞步骤外包；
- 不允许多个角色修改同一片写入范围；
- 最终对变更范围和验证结果负责。

### 3.2 平台边界负责人

负责先判定任务属于哪一层，优先阅读：

- `README.md`
- `docs/DOCS_INDEX.md`
- `docs/system/SYSTEM_MANAGEMENT.md`
- `docs/auth/AUTH_SECURITY.md`
- `docs/auth/AUTH_SESSION_STRATEGY.md`
- `docs/tenant/TENANT_INITIALIZATION.md`

适合处理：

- “这个问题到底属于认证、租户、系统管理还是前端展示？”
- “改这里会不会破坏租户隔离、权限刷新、初始化链路？”

### 3.3 前端壳层与状态负责人

重点范围：

- `frontend/src/App.tsx`
- `frontend/src/components/`
- `frontend/src/shared/components/ViewManager.tsx`
- `frontend/src/stores/systemStore.ts`
- `frontend/src/stores/uiStore.ts`
- `frontend/src/stores/languageStore.ts`
- `frontend/src/modules/auth/store/authStore.ts`

适合处理：

- 登录后初始化；
- 菜单树、标签页、动态视图挂载；
- 权限刷新后的前端重建；
- 全局状态失配；
- 主题、语言、全局壳层行为。

### 3.4 系统管理领域负责人

重点范围：

- `frontend/src/modules/system/`
- `backend/internal/modules/system/`

适合处理：

- 用户、部门、岗位、角色、权限、菜单；
- 字典、日志、系统设置、系统监控；
- system 模块内部接口与页面对齐。

### 3.5 认证与会话负责人

重点范围：

- `frontend/src/modules/auth/`
- `backend/internal/modules/auth/`
- `backend/internal/shared/middleware/`
- `docs/auth/`

适合处理：

- 登录、退出、2FA、API Key；
- `JWT + Refresh Token + Redis` 会话链路；
- 权限刷新、会话失效、强制踢出；
- 当前用户与权限重建。

### 3.6 租户生命周期负责人

重点范围：

- `frontend/src/modules/tenant/`
- `backend/internal/modules/tenant/`
- `backend/internal/shared/database/`
- `docs/tenant/`

适合处理：

- 租户开通、初始化、状态检查；
- Tenant DB 连接与迁移；
- 租户上下文切换；
- 默认数据、默认角色、默认菜单注入。

### 3.7 基础设施与装配负责人

重点范围：

- `backend/internal/app/app_bootstrap.go`
- `backend/internal/shared/database/`
- `backend/internal/shared/cache/`
- `backend/internal/shared/authorization/`
- `backend/internal/shared/i18n/`

适合处理：

- 模块装配；
- 中间件顺序；
- Casbin、Redis、I18n、数据库管理器接线；
- 需要全局生效的基础设施问题。

### 3.8 验证负责人

负责：

- 选择与改动范围相称的最小验证；
- 前端执行 `npm run type-check`、`npm run build`、必要的 UI 冒烟；
- 后端执行模块级或全量 `go test`；
- 记录没有执行的验证和原因。

## 4. 任务分配原则

### 4.1 先按主链路拆，再按技术层拆

优先问：

- 这个问题首先卡在 `auth`、`tenant` 还是 `system`？
- 它会不会穿过前端壳层状态？
- 它会不会触发后端装配或鉴权链路？

不要先问：

- “这是前端问题还是后端问题？”

### 4.2 关键路径留给主协调

以下内容默认不要外包：

- 最终边界判断；
- 是否改数据模型、鉴权链路、初始化链路；
- 最终集成与验收；
- 冲突决策。

### 4.3 只并行独立子任务

可以并行的典型任务：

- 一人读文档，一人查前端入口，一人查后端路由；
- 一人改 `frontend/src/modules/system/`，一人改 `backend/internal/modules/system/`；
- 一人做实现，一人做验证。

不应并行的任务：

- 多人同时编辑同一文件；
- 主协调已经在做的关键阻塞步骤；
- 没有明确写入边界的“大家一起查”。

### 4.4 写入边界必须显式

每个执行角色都要明确：

- 负责哪些目录；
- 不负责哪些目录；
- 输出什么结果；
- 是否允许修改测试和文档。

## 5. 本仓库的默认触发矩阵

### 5.1 菜单、权限、标签页、动态视图问题

默认拉入：

- 主协调
- 前端壳层与状态负责人
- 系统管理领域负责人

按需拉入：

- 认证与会话负责人

必须检查：

- `authStore`
- `systemStore`
- `uiStore`
- `ViewManager`
- `frontend/src/modules/system/views/`
- `backend/internal/modules/system/menu/`
- `backend/internal/modules/system/role/`

### 5.2 登录、刷新、2FA、会话失效问题

默认拉入：

- 主协调
- 认证与会话负责人
- 前端壳层与状态负责人

按需拉入：

- 租户生命周期负责人
- 基础设施与装配负责人

### 5.3 租户初始化、开通、数据库连接问题

默认拉入：

- 主协调
- 租户生命周期负责人
- 基础设施与装配负责人

按需拉入：

- 系统管理领域负责人
- 认证与会话负责人

### 5.4 纯 system CRUD 页面问题

默认拉入：

- 主协调
- 系统管理领域负责人

按需拉入：

- 前端壳层与状态负责人

### 5.5 跨模块全局异常

例如：

- 启动失败；
- 中间件顺序错误；
- I18n 全局失效；
- 租户上下文丢失；
- 权限全部失效。

默认拉入：

- 主协调
- 基础设施与装配负责人
- 对应主线负责人

## 6. 标准执行流程

### 6.1 第一步：定边界

至少回答四个问题：

- 用户真正要什么？
- 属于哪条主线？
- 会影响哪些目录？
- 最小验收标准是什么？

### 6.2 第二步：做代码定位

至少找到：

- 前端入口；
- 后端入口；
- 相关文档入口；
- 需要验证的最短路径。

### 6.3 第三步：拆角色

每个角色都要有：

- 目标；
- 范围；
- 输出；
- 是否允许修改代码。

### 6.4 第四步：并行推进

并行只能发生在：

- 互不冲突的读代码任务；
- 明确分开的写入目录；
- 实现与验证并行。

### 6.5 第五步：主协调集成

主协调必须统一判断：

- 改动是否符合架构；
- 是否引入第二套机制；
- 是否只修表层未修根因；
- 是否需要补文档。

### 6.6 第六步：输出结果

默认输出四件事：

- 改了什么；
- 为什么这样改；
- 做了哪些验证；
- 还剩哪些风险或后续建议。

## 7. 团队设计的具体约束

### 7.1 不按“前后端”粗暴切任务

本仓库很多问题的真实切分应该是：

- 认证链路；
- 租户链路；
- 系统管理链路；
- 前端壳层状态链路；
- 基础设施装配链路。

### 7.2 不复制排查工作

一个角色已经确认的代码结论，其他角色默认复用，不重复做同一轮搜索。

### 7.3 不让多人同时碰同一写入范围

例如：

- `frontend/src/stores/languageStore.ts` 只能由一个角色负责；
- `backend/internal/app/app_bootstrap.go` 只能由一个角色负责；
- 同一 system 子模块目录尽量由一个角色收口。

### 7.4 文档与代码一起收口

以下场景默认补文档：

- 改了主链路；
- 改了模块边界；
- 改了接入方式；
- 改了默认协作流程。

## 8. 推荐的 Team 模式提示模板

可以直接这样发起任务：

```text
按 team 模式处理这个需求：

目标：
影响范围：
验收标准：
已知线索：
限制条件：
```

如果希望更强约束，可以补充：

```text
按 team 模式处理这个需求：

目标：
影响范围：
验收标准：
已知线索：
限制条件：

请先做：
1. 边界判断
2. 角色拆分
3. 根因定位
4. 实施
5. 验证
6. 汇总风险
```

---

## 9. 最小落地要求

如果后续默认采用 Team 模式，至少要坚持以下要求：

- 任何任务先做边界判断；
- 涉及 `menu`、`permission`、`tab`、`view` 时，同步检查前端壳层状态链路；
- 涉及 `login`、`refresh`、`2FA`、`session` 时，同步检查认证链路；
- 涉及 `tenant` 时，同步检查数据库、上下文、初始化链路；
- 最终必须有验证，不允许只给猜测性结论。
