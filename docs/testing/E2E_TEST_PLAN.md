# E2E 测试计划

> 本文定义 Pantheon Platform 当前的端到端测试范围。
> 目标是从真实用户旅程验证“前端壳层 + 后端接口 + 权限/租户链路”是否协同正常。

## 1. 当前工具与入口

- 测试框架：`Playwright`
- 主要目录：`frontend/tests/`
- 默认正式套件：`frontend/tests/official/`
- 手工调试套件：`frontend/tests/manual/`
- 当前正式回归用例：
  - `frontend/tests/official/system-management-functional.spec.ts`
  - `frontend/tests/official/api-key-security.spec.ts`

当前约定：

- `npm run test:e2e` 只执行正式套件；
- 调试/排障脚本不进入默认回归；
- 过时或替代后的试验脚本在确认无引用后直接删除，避免长期堆积。
- 正式 CI 当前拆为系统主流程与 API Key 安全两个专项 workflow。

## 2. E2E 目标

E2E 重点验证：

- 关键用户流程能走通；
- 菜单、权限、标签页、动态视图能协同；
- 认证、租户、系统初始化链路一致；
- 关键运营动作在 UI 上有正确反馈；
- 主流程改动后可快速回归。

## 3. 必测主链路

### 3.1 认证

- 用户登录；
- 登录失败提示；
- 刷新后状态恢复；
- 主动退出；
- 平台管理员特殊登录路径；
- 必要时补 2FA 流程。

### 3.2 租户

- 登录后租户状态检查；
- 首次登录进入初始化向导；
- 测试连接与初始化提交；
- 初始化完成后进入正式工作台；
- 当前租户摘要刷新。

### 3.3 系统管理

核心页面至少覆盖：

- 用户；
- 角色；
- 部门；
- 岗位；
- 菜单；
- 权限；
- 数据字典；
- 系统设置；
- 统一日志；
- 系统监控；
- 租户管理。

### 3.4 工作台运行时

- 侧边栏菜单显示；
- URL 与页面同步；
- 标签页打开/关闭/激活；
- 入口按钮通过统一视图管理跳转；
- 权限变化后工作台重建。

### 3.5 个人安全能力

- 当前已纳入：

- API Key 作用域限制；
- API Key IP 白名单；
- API Key 速率限制。

建议继续纳入：

- 会话管理；
- 登录历史；
- 2FA 开关；
- 当前用户资料刷新。

### 3.6 通知

建议逐步纳入：

- 通知模板创建；
- 发送通知；
- 收件箱已读/未读；
- 统计刷新。

## 4. 数据准备原则

- 优先使用 API 辅助构造测试数据；
- 避免 UI 链路互相依赖导致用例脆弱；
- 高风险清理动作优先走 API 或专用脚本；
- 多租户场景要显式区分平台租户与测试租户。

## 5. 用例设计原则

- 一条用例只覆盖一条主旅程；
- 复杂主流程可在 API 层做前置准备；
- 优先校验用户可见结果，不只盯接口码；
- 高风险动作必须验证确认弹层和结果反馈；
- 中文与英文文案至少做基础可见性校验。

## 6. 通过标准

发布前的 E2E 至少应满足：

- 核心登录与系统管理主流程通过；
- 关键导航与权限显示无明显回归；
- 新增功能的主用户旅程有对应自动化或明确手工用例；
- 失败用例可定位到明确链路。

## 7. 常用命令

```bash
cd frontend
npm run test:e2e
npm run test:e2e:system
npm run test:e2e:api-key
npm run build
```

如需手工排障：

```bash
cd frontend
npm run test:e2e:manual
```

## 8. 当前缺口

当前 E2E 已形成“系统主流程 + API Key 安全专项”双基线，仍建议继续补：

- 2FA 全链路；
- 会话管理全链路；
- 通知中心；
- 权限拒绝矩阵；
- 多租户切换与租户停用联动；
- 白屏、错误边界和异常态回归。
