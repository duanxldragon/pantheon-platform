# 白盒测试规范

> 本文定义 Pantheon Platform 的白盒测试口径。  
> 白盒测试不是“只要调用函数就算”，而是基于真实实现边界验证核心分支、失败路径和约束条件。

## 1. 适用范围

适用于：

- 后端 `service / middleware / authorization / tenant / notification`；
- 前端 `store / hook / util / 运行时协调逻辑`；
- 需要验证权限、租户、状态机、事务、重试、异常分支的改动。

## 2. 核心原则

- 测业务规则，不只测返回值；
- 测失败路径，不只测 happy path；
- 测边界条件，不只测正常数据；
- 测协作约束，不只测单函数；
- 保持小范围、快反馈、可复现。

## 3. 后端白盒重点

### 3.1 必测类型

- 权限判断与数据范围；
- 事务回滚；
- 批量操作原子性；
- 跨租户拒绝；
- `auth_version` / `revoked_after`；
- Redis / DB 协作失败分支；
- 幂等性与重复调用保护；
- 审计摘要与错误包装。

### 3.2 推荐落点

- `*_service_test.go`
- `*_handler_test.go`
- `*_batch_test.go`
- `*_middleware_test.go`

### 3.3 典型场景

- 用户越权读取/写入被拒绝；
- 角色、菜单、权限变更后联动刷新；
- 租户外数据被正确过滤；
- 批量删除在中途失败时回滚；
- 通知任务失败后保留正确失败语义。

## 4. 前端白盒重点

### 4.1 必测类型

- `authStore` 状态迁移；
- `systemStore.initialize()` 聚合行为；
- `api_client` / `axios_client` 的刷新与 401 处理；
- 权限判断工具；
- 路由守卫与运行时协作；
- Hook 的派生状态与副作用。

### 4.2 推荐落点

- `*.test.ts`
- `*.test.tsx`

优先测试：

- store；
- hook；
- util；
- 小型状态型组件。

不建议把纯视觉快照当成主要白盒手段。

## 5. 测试设计要求

每个白盒测试尽量至少回答其中两项：

- 正常路径是否成立；
- 非法输入是否被拒绝；
- 上下文缺失是否 fail closed；
- 权限/租户边界是否守住；
- 副作用是否正确触发；
- 错误时是否保持一致状态。

## 6. 命名建议

- `TestBatchDeleteRejectsOutOfScopeUsers`
- `TestRefreshTokenPreservesSessionID`
- `auth_store.test.ts`
- `use_route_guard.test.ts`

名称应体现：

- 触发条件；
- 被验证规则；
- 期望行为。

## 7. 完成定义

白盒测试补充后，应具备：

- 新增规则至少一条正常路径；
- 至少一条失败或边界路径；
- 涉及租户/权限/事务时必须覆盖对应约束；
- 不依赖外部不稳定环境才能稳定执行。

## 8. 与其他层级的关系

- 白盒测试负责证明“实现规则正确”；
- 集成测试负责证明“模块协作正确”；
- E2E 负责证明“用户旅程正确”。

不要用 E2E 替代白盒，也不要只靠白盒宣布功能完成。
