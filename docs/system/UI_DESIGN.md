# Pantheon Platform UI 设计规范

> 本文定义 Pantheon Platform 的平台级 UI 设计边界。  
> 它回答“平台应该呈现什么气质、如何建立一致性、不同页面家族如何协作”，不展开具体组件源码与目录细节。  
> 面向前端实现与 AI 生成的稳定设计描述，统一看 `frontend/DESIGN.md`；具体落地基座看 `frontend/docs/system/UI_IMPLEMENTATION_GUIDE.md`。  
> 当前稳定风格额外吸收了 `awesome-design-md/design-md/figma/DESIGN.md` 中“排版驱动、壳层中性、色彩克制”的专业特征。
> 若需要页面模板、交付口径和当前改造完成面，继续看 `frontend/docs/system/UI_PAGE_TEMPLATES.md`、`frontend/docs/system/UI_DELIVERY_CHECKLIST.md`、`frontend/docs/system/UI_REFACTOR_SUMMARY.md`。

## 1. 设计目标

Pantheon Platform 是多租户后台平台，不是品牌宣传站，也不是纯 CRUD 页面集合。  
UI 设计需要同时支撑以下三种体验：

- **工作台稳定性**：菜单、标签页、动态视图在长时间使用下仍然清晰、低噪、可恢复；
- **系统管理效率**：高频列表、筛选、表单、详情操作要优先保证识别效率；
- **平台扩展一致性**：新模块接入时，不需要重新发明一套视觉和交互语言。

因此平台整体风格应保持：

- 专业、克制、可信；
- 信息密度中高，但不压迫；
- 强语义弱装饰；
- 亮暗主题都能稳定工作；
- 对中文与英文混排、长字段、动态权限状态有韧性。

本文中的关键术语默认含义如下：

- **工作台壳层**：菜单、顶部栏、标签页、主视图共同构成的运行时外壳；
- **管理页骨架**：标题区、筛选区、操作区、结果区的标准页面结构；
- **只读详情链路**：摘要、状态、审计、原文分组组成的详情查看方式；
- **结果反馈**：成功态、确认态、巡检态等页内反馈，不只指 toast；
- **语义 token**：通过角色表达用途的设计令牌，而不是具体颜色名。

可以明确借鉴的专业特征：

- 壳层更中性，色彩更节制；
- 层级更多依赖排版、边框、间距，而不是大面积彩色块；
- 控件几何保持统一，不在页面间频繁变形；
- 焦点态与当前态有设计辨识度。

## 2. 视觉基调

### 2.1 Atmosphere

平台推荐的整体气质为：

- 以中性背景和语义强调色为主；
- 容器分层清晰，但避免过重阴影和过强渐变；
- 重点强调“可扫描”“可判断”“可恢复”，而不是“强展示感”；
- 动效存在，但只服务于状态变化与层级提示。

### 2.2 页面密度原则

不同页面家族采用不同密度：

| 页面家族 | 典型页面 | 密度策略 |
| :--- | :--- | :--- |
| 工作台壳层 | 菜单、顶部栏、标签页 | 低干扰、弱装饰、稳定对齐 |
| 管理列表页 | 用户、角色、菜单、日志 | 中高密度，强调对齐、分组、扫描效率 |
| 表单与详情页 | 新增、编辑、查看详情 | 中密度，强调输入节奏与阅读顺序 |
| 初始化/空态页 | 登录、租户初始化、空结果 | 中低密度，强调引导与下一步 |

## 3. 设计令牌原则

### 3.1 Token 优先

平台所有颜色、边框、阴影、圆角、焦点、图表系列色，都应优先通过语义化 token 表达，而不是在业务页面里直接写具体色值或具体色阶类名。

核心原则：

- 用“语义”命名，而不是用“颜色”命名；
- 组件消费 token，不消费品牌稿中的临时值；
- 亮色与暗色通过 token 切换，不在业务页手写两套主题；
- 新模块接入时优先复用既有 token，不再派生第三套视觉系统。

### 3.2 语义角色

平台至少应稳定维护以下语义角色：

| 角色 | 用途 |
| :--- | :--- |
| `background` / `foreground` | 页面底色与正文文字 |
| `card` / `card-foreground` | 卡片、面板、详情区 |
| `popover` / `popover-foreground` | 下拉、弹层、悬浮层 |
| `primary` / `primary-foreground` | 主动作、主强调 |
| `secondary` | 次级操作、弱强调 |
| `muted` / `muted-foreground` | 次要信息、占位、说明 |
| `accent` | 当前项强调、悬停强调 |
| `destructive` | 删除、不可逆、危险操作 |
| `border` / `input` / `ring` | 边框、输入态、聚焦态 |
| `success` / `warning` / `info` | 状态反馈、标签与提示 |

## 4. 版式与层级

### 4.1 间距

- 统一遵守 `8px` 步进体系；
- 页面主内容区默认使用稳定的横向留白；
- 列表筛选区、操作区、结果区之间应有清晰的节奏分隔；
- 不要在同一页面混入多套不相关的间距规则。

### 4.2 圆角

平台圆角分三档即可：

| 档位 | 适用对象 |
| :--- | :--- |
| 小圆角 | 输入框、按钮、下拉、徽标 |
| 中圆角 | 卡片、筛选区、列表容器 |
| 大圆角 | 对话框、抽屉、重点容器 |

约束：

- 输入类组件保持精确；
- 容器类组件保持包裹感；
- 弹层类组件明显区别于页内容器。

### 4.3 深度

深度表达依次通过以下顺序建立：

1. 背景色差；
2. 细边框；
3. 小幅阴影；
4. 仅对顶层弹层使用更强阴影。

禁止：

- 用大面积毛玻璃覆盖数据密集区；
- 用夸张阴影和强渐变取代层级结构；
- 在管理页滥用营销风格光效。

## 5. 核心组件家族

### 5.1 导航壳层

导航壳层包含：

- 侧边菜单；
- 顶部栏；
- 面包屑；
- 标签页；
- 主视图容器。

设计要求：

- 当前激活态要明确，但不刺眼；
- 菜单、标签页、主视图三者的层级关系稳定；
- 权限变更、菜单刷新后，界面不应出现“视觉断层”；
- 壳层风格要长期稳定，避免频繁改版。
- 壳层更接近专业工具软件的中性 chrome，而不是彩色看板。

### 5.2 列表页

标准列表页应遵守：

- 标题区、筛选区、操作区、结果区分层明确；
- 表格表头清晰，列宽策略兼顾中文与英文；
- 状态列优先使用语义标签，不用纯文案硬拼；
- 行内操作超过 3 个时应收纳；
- 批量操作只在选中后显性出现。

### 5.3 表单页

标准表单页应遵守：

- 表单分组明确；
- 必填、错误、说明、禁用态可快速识别；
- 高风险提交需要二次确认；
- 长表单优先分段，不堆成长墙；
- 弹层表单和页面表单保持同一套字段语义。

### 5.4 详情页

详情页重点是“扫读效率”而不是“视觉花样”：

- 头部摘要先回答对象是谁、当前状态如何；
- 详情内容按模块分组；
- 操作入口靠近对象状态；
- 审计信息、系统字段放在弱化区而不是主阅读区。

### 5.5 结果反馈

结果反馈属于平台级体验的一部分，默认包括：

- 提交前确认；
- 提交成功摘要；
- 巡检结论；
- 空态与下一步动作；
- 风险提示与对象级确认。

设计要求：

- 结果反馈优先落在当前上下文中，而不是只依赖全局 toast；
- 反馈要说明“发生了什么、影响了什么、下一步做什么”；
- 反馈卡和页面主体保持同一套容器语言，不单独造一套横幅风格；
- 成功态保持克制，危险态保持明确。

## 6. 反馈与状态

### 6.1 加载

- 页面初始化可使用全局加载态；
- 列表、卡片、详情页优先使用骨架屏；
- 禁止在已知结构页面长期使用全屏 Spinner；
- 骨架屏应尽量贴近真实布局。

### 6.2 空态

空态必须告诉用户三件事：

1. 当前为什么为空；
2. 是否是筛选导致；
3. 下一步可以做什么。

### 6.3 错误与风险

- 普通错误使用轻量提示；
- 阻断性错误使用明确的说明区或弹层；
- 删除、停用、重置、会话踢出等高风险动作必须视觉分层清晰；
- 危险按钮文案应指向具体对象与结果。

## 7. 响应式与国际化韧性

### 7.1 响应式

平台优先保证桌面端后台体验，同时兼顾中小屏：

- 超宽表格允许横向滚动；
- 必要时冻结首列或关键标识列；
- 中小屏应主动折叠次要字段；
- 极简数据结构可切换卡片视图。

### 7.2 国际化

- 文案长度不可假设永远固定；
- 列、按钮、标签、标题要为英文长词保留空间；
- 关键截断文本应配套 Tooltip；
- 中文、英文、数字混排时仍需对齐稳定。

## 8. 可访问性基线

平台最低要求：

- 颜色对比符合基本可读性；
- 所有交互控件都有清晰焦点态；
- 弹层支持键盘关闭与焦点约束；
- 状态信息不能只依赖颜色传达；
- 可点击区域保持足够尺寸。

## 9. 设计守则

### 9.1 Do

- 优先复用统一页面模板；
- 优先做语义一致性，再做视觉强化；
- 把动态菜单、权限、标签页视为同一工作台系统；
- 在亮暗主题下都检查边框、文字、禁用态和空态。

### 9.2 Don't

- 不要把后台管理页做成营销落地页；
- 不要在页面里硬编码品牌色和灰阶；
- 不要新造第二套按钮、表格、弹层、筛选容器规范；
- 不要为了“高级感”牺牲扫描效率与状态辨识；
- 不要让工作台壳层和系统页风格割裂。

## 11. 验收清单

- [ ] 新页面是否沿用统一 token，而不是硬编码颜色与阴影？
- [ ] 列表、表单、详情三类页面是否使用统一结构节奏？
- [ ] 危险操作是否具备明确的视觉与文案分层？
- [ ] 亮色与暗色主题下是否都可读、可点、可判断？
- [ ] 菜单、标签页、动态视图的状态切换是否保持一致体验？
