# 系统管理模块设计

## 1. 模块定位

系统管理模块是平台在“租户初始化完成之后”最先进入的核心管理域，负责解决租户内部的基础治理问题：

- 管人：用户、部门、岗位；
- 管权：角色、权限、菜单；
- 管过程：登录后初始化、审计日志、系统设置、系统监控；
- 管个人：个人中心、安全设置、个人偏好；
- 管扩展：为后续业务模块提供统一接入底座。

它不是某个业务线自己的功能模块，而是所有后续业务模块共享的管理底盘。

与其他专题的边界如下：

- `README.md`：只讲平台总体设计与文档入口；
- `docs/auth/AUTH_SECURITY.md`：讲登录、会话、2FA、安全策略；
- `docs/auth/AUTH_SESSION_STRATEGY.md`：讲权限刷新与强制失效机制；
- `docs/tenant/TENANT_INITIALIZATION.md`：讲租户开通、初始化、生命周期；
- 本文：只讲系统管理模块自身的业务模型、对象关系、交互规则与扩展规范。

---

## 2. 业务边界

### 2.1 系统管理模块负责

- 用户管理；
- 部门管理；
- 岗位管理；
- 角色管理；
- 权限管理；
- 菜单管理；
- 日志管理；
- 系统设置；
- 系统监控；
- 个人中心；
- 数据字典等系统级支撑能力。

### 2.2 系统管理模块不直接负责

- 租户创建、租户停用、租户数据库初始化；
- 登录凭证签发、Refresh Token、2FA 校验；
- 某个具体业务对象本身的领域逻辑，例如主机、应用、工单、资产、流程；
- 部署、发布、运维编排等交付动作。

这几个边界一定要明确：

- 租户管理负责“开租户、配租户、管租户生命周期”；
- 认证模块负责“让谁进来、会话何时失效”；
- 系统管理负责“进来以后怎么管理组织、账号、权限、菜单与基础配置”；
- 业务模块负责“自己的领域对象与业务流程”。

## 3. 核心对象与关系模型

系统管理模块围绕六类核心对象运转：用户、部门、岗位、角色、权限、菜单。

### 3.1 关系总览

| 对象 | 关系 | 说明 |
| :--- | :--- | :--- |
| 用户 → 部门 | 多对一 | 一个用户归属一个主部门，一个部门下可有多个用户 |
| 用户 → 岗位 | 多对一 | 一个用户绑定一个主岗位，一个岗位可被多个用户复用 |
| 用户 ↔ 角色 | 多对多 | 用户可以拥有多个角色，角色也可以授权给多个用户 |
| 角色 ↔ 权限 | 多对多 | 角色决定用户可执行的接口、按钮、数据范围 |
| 角色 ↔ 菜单 | 多对多 | 角色决定用户登录后可见的导航、入口与页面 |
| 菜单 ↔ 前端组件 | 一对一映射 | 菜单通过 `component` 标识前端实际页面 |
| 部门 → 部门 | 一对多 | 支持树形组织结构 |

### 3.2 关系的业务含义

- **部门** 解决组织归属、上下级结构、负责人、数据归口问题；
- **岗位** 解决岗位职责、职位归类、审批或责任分工问题；
- **角色** 解决授权聚合与权限组合问题；
- **权限** 解决“能做什么”的问题；
- **菜单** 解决“能看到什么、能进入哪里”的问题；
- **用户** 是最终的权限承载主体。

### 3.3 设计原则

- 权限尽量绑定到角色，而不是直接绑定到用户；
- 菜单可见性由角色控制，不建议手工给单个用户单独配菜单；
- 部门与岗位主要承载组织语义，不替代角色授权；
- 用户最终可见页面、按钮与数据范围，是多个角色聚合后的结果；
- 所有关联关系都必须在当前租户内闭环，不能跨租户串用。

## 4. 子模块详细说明

### 4.1 用户管理

用户管理是系统管理模块的核心入口，承担账号主档维护。

#### 负责内容

- 创建、编辑、删除用户；
- 启用、禁用用户；
- 绑定部门、岗位；
- 分配一个或多个角色；
- 修改密码、管理员重置密码；
- 查看最近登录时间、登录 IP、账号状态；
- 维护头像、手机号、邮箱、昵称等基础资料。

#### 关键业务规则

- 禁用用户后，用户不可继续登录；
- 删除用户前，需要清理角色关系与相关引用；
- 修改密码、重置密码属于安全事件，需要让旧会话失效；
- 用户角色变更后，当前登录用户的授权结果需要刷新；
- 用户归属部门、岗位变化后，相关数据范围与个人上下文需要更新。

#### 与其他对象的交互

- 用户分配角色后，角色上的权限和菜单才真正生效；
- 用户属于哪个部门，会影响组织树与数据可见范围；
- 用户属于哪个岗位，会影响岗位视角的筛选与业务协作；
- 用户个人中心读取的也是用户主档，只是入口在右上角。

### 4.2 部门管理

部门管理承载组织结构，是数据权限和管理归属的重要基础。

#### 负责内容

- 树形组织结构维护；
- 新增、编辑、删除部门；
- 设置父级部门；
- 设置部门负责人；
- 统计部门下成员。

#### 关键业务规则

- 删除部门前，需要校验是否存在子部门；
- 删除部门前，需要校验是否仍挂载用户；
- 部门负责人必须属于当前租户；
- 组织树一旦变化，涉及部门范围的数据权限应同步生效。

### 4.3 岗位管理

岗位管理用于表达职位属性与职责归类，是组织语义的补充。

#### 负责内容

- 岗位新增、编辑、删除；
- 岗位编码、名称、状态维护；
- 与用户建立岗位归属关系；
- 为后续审批、责任人、值班等业务提供基础字段。

#### 关键业务规则

- 删除岗位前，需要确认是否仍被用户引用；
- 岗位变更通常不直接决定权限，但会影响个人上下文与业务规则；
- 岗位调整后，用户当前登录态的展示信息应尽快刷新。

### 4.4 角色管理

角色管理是权限系统的中心节点。

#### 负责内容

- 角色新增、编辑、删除；
- 角色启用、停用；
- 角色分配给用户；
- 角色绑定权限；
- 角色绑定菜单；
- 角色配置数据范围。

#### 关键业务规则

- 角色删除时，需要同步清理用户绑定关系、权限关系、菜单关系；
- 系统预置角色建议限制删除；
- 角色停用后，该角色产生的授权结果不应继续生效；
- 角色变更后，所有拥有该角色的在线用户都需要重新加载授权结果。

### 4.5 权限管理

权限管理解决“用户能做什么”的问题。

#### 权限类型

- API 权限：控制接口访问；
- 按钮权限：控制前端按钮、操作入口显示；
- 数据权限：控制数据范围；
- 可扩展权限：如字段权限、资源级权限。

#### 关键业务规则

- 权限码必须稳定、唯一、可读；
- 权限最好按模块统一命名，例如：
  - `system:user:view`
  - `system:user:create`
  - `system:role:assign`
  - `host:machine:restart`
- 权限变更以角色为传播中心，不直接对单用户做大规模离散授权；
- 权限变更后，需要让在线用户尽快拿到新的授权结果。

### 4.6 菜单管理

菜单管理解决“用户能看到什么、能进入哪里”的问题。

#### 菜单模型

推荐采用三级结构：

- 目录：只承载分组与导航；
- 菜单：承载具体页面入口；
- 按钮：承载页面内操作点或辅助权限节点。

#### 负责内容

- 菜单树维护；
- 菜单路径、名称、图标、排序、状态维护；
- 菜单与前端 `component` 映射维护；
- 菜单与角色绑定；
- 页面可见性与动态挂载的基础配置。

#### 关键业务规则

- 菜单树必须保持可组装的父子关系；
- 菜单 `component` 是前端动态加载页面的关键字段；
- 角色只要绑定了菜单，用户登录后就应能看到对应入口；
- 菜单名称、路径、组件、状态变化后，在线用户菜单树应刷新；
- 删除菜单时，需要同步处理角色菜单关系与前端可见性。

### 4.7 日志管理

日志管理承载平台审计能力，是系统可追踪性的基础。

#### 负责内容

- 操作日志；
- 登录日志；
- 条件检索、分页、导出、清理；
- 关键安全事件留痕。

#### 审计要求

- 关键写操作必须有操作日志；
- 登录成功、失败、退出、锁定、改密、2FA 等安全事件必须留痕；
- 后续业务模块新增高风险操作时，应无条件接入审计链路。

### 4.8 系统设置

系统设置承载租户内系统级配置，不承担租户生命周期管理。

#### 负责内容

- 基础键值配置；
- 邮件配置；
- 短信配置；
- 功能开关；
- 第三方接入参数；
- 业务模块公共参数。

#### 关键业务规则

- 设置项应区分平台级、租户级、个人级边界；
- 涉及密钥、密码的设置项必须加密保存；
- 高风险配置变更应记录审计日志；
- 配置变更后，按需决定是否需要即时生效或重载。

### 4.9 系统监控

系统监控负责展示平台运行状态，不直接替代专业监控系统。

#### 负责内容

- 主库连接状态；
- 租户库连接状态；
- Redis 状态；
- 在线用户统计；
- 服务运行时信息；
- 模块健康检查入口。

#### 设计目标

- 给管理员快速判断“系统是否可用”的能力；
- 给后续业务模块暴露统一健康状态接口；
- 给告警与通知提供接入点。

### 4.10 个人中心

个人中心是系统管理面向当前登录用户的自服务入口。

#### 负责内容

- 个人资料维护；
- 头像更新；
- 密码修改；
- 2FA 绑定、验证、关闭；
- API Key 管理；
- 登录历史查看；
- 语言、主题、通知等个人偏好。

#### 设计原则

- 个人中心不绕过系统权限模型；
- 安全相关动作要走认证模块能力；
- 个人偏好是用户级配置，不应污染租户级系统设置。

### 4.11 数据字典

数据字典属于系统级支撑域，用于统一维护状态枚举、类别选项、展示标签等基础数据。

它不是权限中心，但会被系统管理和后续业务模块广泛复用。

#### 数据模型

数据字典分两层：

| 层级 | 模型 | 表名 | 说明 |
| :--- | :--- | :--- | :--- |
| 字典类型 | `DictType` | `system_dict_types` | 定义一组枚举的分类，例如"用户状态"、"性别" |
| 字典数据 | `DictData` | `system_dict_data` | 某个类型下的具体条目，例如"启用/禁用" |

`DictType` 核心字段：

- `code`：类型唯一编码（租户内唯一），业务模块通过 code 引用，例如 `user_status`
- `name`：类型展示名
- `status`：`active` / `inactive`
- `tenant_id`：租户隔离

`DictData` 核心字段：

- `type_id`：归属字典类型 ID
- `label`：展示标签（例如"启用"）
- `value`：机器读取值（例如 `active`）
- `sort`：排序权重
- `status`：`active` / `inactive`

#### 关键业务规则

- 字典数据在租户内隔离，`type_id` 和 `tenant_id` 共同确定唯一性
- 业务模块引用字典时应通过 `code` 而非 ID，`code` 是稳定的语义标识
- 删除字典类型前需确认没有字典数据引用；删除字典数据前应确认是否被业务数据引用
- 停用的字典数据在查询接口中可根据需要过滤，但不建议物理删除已被业务引用的条目
- 字典数据变更不触发会话刷新，但可能需要业务方缓存清理

#### 适用场景

- 表单的下拉选项（用户状态、性别、优先级等）
- 状态标签的展示映射（后端存 `active`，前端展示"启用"）
- 业务枚举的动态化（不需要发版就能调整选项）
- 后续业务模块可以注册自己的字典类型，通过系统管理界面统一维护

## 5. 登录后的权限初始化

系统管理模块的核心特点之一，是登录成功后并不是直接进入静态页面，而是要完成一轮运行时初始化。

### 5.1 初始化目标

登录后至少要完成以下事情：

1. 识别当前用户与当前租户；
2. 获取当前用户信息；
3. 获取当前角色集合；
4. 聚合角色对应的权限集合；
5. 聚合角色对应的菜单集合；
6. 初始化页面导航、首页、标签页与按钮权限；
7. 初始化系统管理相关基础数据。

### 5.2 为什么必须动态初始化

因为以下因素都可能在运行期变化：

- 用户角色变化；
- 角色权限变化；
- 角色菜单变化；
- 菜单组件配置变化；
- 部门与岗位关系变化；
- 不同租户拥有不同菜单与授权范围；
- 新业务模块可能在不改主框架的情况下动态接入。

所以权限、菜单和页面挂载必须采用运行时动态装配，而不能完全写死在前端静态路由中。

### 5.3 初始化结果

初始化完成后，前端至少应得到三类结果：

- **用户上下文**：我是谁、属于哪个租户、有哪些角色；
- **授权结果**：我有哪些权限、能执行哪些动作；
- **导航结果**：我能看到哪些菜单、进入哪些页面。

## 6. 变更传播与会话协同

系统管理模块不是只管“增删改查”，它还要对已登录用户的在线体验负责。

### 6.1 需要触发授权刷新的场景

以下变更发生后，应让当前在线用户重新拉取授权结果：

- 用户角色变更；
- 角色权限变更；
- 角色菜单变更；
- 菜单名称、路径、组件、状态变化；
- 部门变化；
- 岗位变化。

### 6.2 需要触发强制失效的场景

以下属于安全或租户状态事件，应让当前会话直接失效：

- 用户被停用；
- 用户被删除；
- 用户修改密码；
- 管理员重置密码；
- 租户被停用；
- 租户被删除。

这里不重复描述底层 Redis Key 与中间件机制，统一以 `docs/auth/AUTH_SESSION_STRATEGY.md` 为准。

### 6.3 模块协同原则

- 系统管理模块负责定义“哪些业务变更会影响授权结果”；
- 认证模块负责执行“如何刷新、如何失效、如何拦截”；
- 前端负责在刷新成功后重建用户、权限、菜单与页面状态。

## 7. 与多租户体系的关系

系统管理模块始终运行在租户上下文内。

### 7.1 租户隔离要求

以下数据默认都是租户内数据：

- 用户；
- 部门；
- 岗位；
- 角色；
- 权限；
- 菜单；
- 日志；
- 设置；
- 数据字典。

### 7.2 租户上下文校验要求

- 用户分配角色时，角色必须属于当前租户；
- 角色绑定权限、菜单时，权限和菜单必须属于当前租户；
- 部门负责人、岗位引用、用户归属都必须校验租户一致性；
- 日志、设置、监控查询都必须基于当前租户过滤。

### 7.3 与部署模式的关系

- 私有化单租户场景下，可以默认绑定平台默认租户；
- PaaS / SaaS 场景下，系统管理模块必须显式依赖租户上下文；
- 不管部署模式如何变化，系统管理模块的对象关系模型都尽量保持一致。

租户开通、数据库初始化、生命周期流转，统一参考 `docs/tenant/TENANT_INITIALIZATION.md`。

## 8. 新业务模块接入规范

系统管理模块要服务于后续业务扩展，因此需要稳定的接入规范。

### 8.1 新模块至少要接入哪些底座能力

- 租户隔离；
- 角色权限；
- 菜单挂载；
- 日志审计；
- 国际化；
- 系统设置；
- 监控与通知。

### 8.2 标准接入步骤

1. 定义租户级业务模型；
2. 定义模块权限码；
3. 定义模块菜单与前端 `component`；
4. 接入后端接口与服务；
5. 接入前端页面与状态管理；
6. 在角色管理中完成菜单与权限授权；
7. 通过登录后动态初始化自动挂载显示。

### 8.3 以主机管理为例

| 接入点 | 建议内容 |
| :--- | :--- |
| 菜单 | 资源管理 / 主机管理 |
| 页面组件 | `host/HostManagement` |
| 权限码 | `host:machine:view`、`host:machine:create`、`host:machine:delete` 等 |
| 审计 | 主机创建、编辑、删除、远程操作必须留痕 |
| 监控 | 主机状态、巡检结果、异常告警可接入统一监控 |
| 国际化 | 页面文案、状态枚举、错误信息统一走多语言资源 |

### 8.4 动态挂载要求

- 后端菜单配置维护 `component`；
- 前端视图映射维护 `component` 到实际页面组件的映射；
- 角色授权决定该菜单是否对用户可见；
- 权限码决定页面按钮是否可用；
- 新业务模块接入时，尽量不要修改平台主导航骨架。

## 9. 设计约束与落地建议

### 9.1 约束

- 先有角色，再有权限聚合，不建议大规模直接对用户授权；
- 菜单是导航配置，不等于权限本身，但会影响页面可见性；
- 部门、岗位主要表达组织语义，不直接替代授权模型；
- 所有变更都要考虑对在线用户的影响；
- 所有系统级能力都要考虑租户隔离与多语言扩展。

### 9.2 建议

- 系统预置最小可用角色、菜单、权限模板；
- 新业务模块统一遵循权限码命名规范；
- 对高风险操作统一补审计日志；
- 对影响在线授权的变更统一纳入刷新或失效策略；
- 文档分工保持稳定：业务规则只在本文维护，实现细节分别落到 `backend/docs/` 和 `frontend/docs/`。

### 9.3 预置角色说明

系统在首次启动时（`bootstrapDefaultData`）会幂等创建以下预置数据：

| 对象 | 说明 | 约束 |
| :--- | :--- | :--- |
| `platform` 租户 | 平台默认租户，ID 为全零 UUID | 不能删除 |
| `super_admin` 角色 | 超级管理员，`type=system`，`data_scope=all`，`is_system=true` | 不能删除，不建议修改 |
| `admin` 用户 | 由 `default_admin.enabled` 控制；生产环境必须关闭 | 生产环境禁止启用 |

**预置角色的权限策略**：

`super_admin` 角色在 Casbin 中被授予以下策略：

```
/api/v1/* → *（所有方法）
/api/v1/tenants/* → *（所有方法）
```

即对所有 API 路径具有完整访问权限。

**租户初始化后的最小可用数据**：

租户初始化完成后，系统管理模块的迁移器会建立必要的表结构，并由租户初始化流程注入最小可用的系统菜单、权限和角色模板。

当前初始化链路默认预置的最小角色模板：

| 角色 | 建议数据范围 | 建议权限集合 |
| :--- | :--- | :--- |
| 租户超级管理员 | `all` | 系统管理全量权限 |
| 普通用户 | `self` | 仅个人中心权限 |
