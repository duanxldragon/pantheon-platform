# Backend Docs AGENTS

> 本文件补充 `../AGENTS.md` 和 `../../AGENTS.md`，只约束 `backend/docs/` 目录下的文档维护行为。

## 1. 文档边界

- 这里只写后端实现视角，不重复平台级业务设计。
- 这里重点回答“后端代码怎么落、链路怎么跑、模块怎么协作”。
- 平台级设计继续维护在 `../../docs/`。
- Swagger 生成产物不在这里维护，`backend/docs/` 只描述其入口和维护方式。

## 2. 写文档前先判断

- 这是后端工程问题，还是平台业务规则问题？
- 这是总览问题，还是专题实现问题？
- 能否优先更新现有文档，而不是新增一份平行文档？

## 3. 默认写法

- 先写目录职责和链路，再写实现约束。
- 先写模块边界，再写扩展方式。
- 避免把文档写成源码逐行翻译。
- 对高频变化的字段、表结构和细枝末节，只给入口和边界，不在总览文档中硬维护。

## 4. 明确禁止

- 不要把 `backend/docs/` 写成 Swagger 替代品。
- 不要把平台级业务规则和后端实现规则混写。
- 不要在多份文档里重复维护同一份目录结构说明。
- 不要为了“完整”而复制大段代码。

## 5. 推荐分工

- `BACKEND_GUIDE.md`：工程总览、启动链路、基础设施、扩展方式
- `docs/BACKEND_DOCS_INDEX.md`：后端文档入口
- `docs/auth/`：认证与安全的后端实现
- `docs/system/`：系统管理模块的后端实现
- `docs/tenant/`：租户与初始化链路的后端实现

## 6. 更新原则

- 改启动链路、模块装配、数据库管理、授权协作时，要同步更新相关文档。
- 改命名规范、分层规则、工具链入口时，也要同步更新文档。
- 入口文档保持短，细节拆到专题文档。

## 7. 提交前

- 说明改的是总览还是专题实现文档。
- 说明是否影响后端分层、链路、模块边界。
- 说明是否还有其他后端文档需要后续同步。
